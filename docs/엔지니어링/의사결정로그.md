# 의사결정 로그 (ADR — 백엔드)

> **누가 봄**: 백엔드 개발자
> **언제 봄**: 큰 결정 직전 (전례 검색) / 결정 직후 (기록)
> **공유**: 일부 결정은 클라이언트 ADR 와 중복. 백엔드 디테일 위주.

## 형식

```
## ADR-XXX — YYYY-MM-DD
**결정**: 한 줄
**대안**: 검토했던 다른 옵션들
**선택 이유**: 왜 이걸 골랐나
**영향**: 다른 코드/설정에 미치는 영향
**되돌리기 비용**: 낮음 / 중간 / 높음
**관련**: PR / 이슈
```

시간 역순 정렬.

---

## ADR-007 — 2026-04-26
**결정**: IP 기반 Rate Limit (Bucket4j in-memory) 도입
**대안**: Redis 분산 limit, Cloudflare WAF, nginx limit_req
**선택 이유**:
- 단일 인스턴스 메모리로 충분
- Bucket4j Spring 통합 단순
- Redis 추가는 비용 + 복잡도
**영향**: scale-out 시 Redis 마이그 필요
**되돌리기 비용**: 낮음
**관련**: PR `feat/rate-limiting`

## ADR-006 — 2026-04-26
**결정**: WS 가격 폴링에 시장 OPEN 가드 + cache TTL 동적
**대안**: 5초 → 30초로 그냥 늘리기
**선택 이유**: 휴장 시간 호출 자체가 무의미. 시장별 분리 검증.
**영향**: `marketSessionService.buildMarketSessions()` 매 5초 호출 (cheap)
**되돌리기 비용**: 낮음
**관련**: PR `chore/cost-savings-railway`

## ADR-005 — 2026-04-25
**결정**: KR 휴장 캘린더 코드 하드코딩 (`KR_HOLIDAYS_BY_DATE` map)
**대안**: 외부 캘린더 API, korean-calendar 라이브러리
**선택 이유**: 음력 + 대체휴일 + 임시공휴일 자동 계산 신뢰성 낮음. KRX 공식 12월 공시 후 수동 갱신 안전.
**영향**: 매년 12월 1회 갱신 절차 추가
**되돌리기 비용**: 낮음

## ADR-004 — 2026-04-25
**결정**: JWT 더미 secret 부팅 시 fail-fast 안 함 (ERROR 로그만)
**대안**: 더미값이면 부팅 거부
**선택 이유**: Railway healthcheck 끊겨서 운영자가 secret 못 넣고 디버깅 불가
**영향**: 운영자가 ERROR 로그 못 보면 보안 취약 상태로 운영
**되돌리기 비용**: 낮음

## ADR-003 — 2026-04-24
**결정**: Google OAuth `aud` (audience) 화이트리스트 도입
**대안**: 검증 안 함 (기존 코드)
**선택 이유**: 다른 GCP 클라이언트 ID 토큰 통과 차단
**영향**: 환경변수 `SIGNAL_DESK_AUTH_GOOGLE_ALLOWED_AUDIENCES` 추가 필수
**되돌리기 비용**: 낮음

## ADR-002 — 2026-04 (초기)
**결정**: 시장 데이터 캐시 in-memory (60s/3m + 5m/15m TTL)
**대안**: Redis, DB 캐시, 캐시 없음
**선택 이유**:
- 단일 인스턴스 단계 — 메모리로 충분
- 외부 API (KRX/Naver/FRED) 호출 비용·차단 위험 절감
**영향**: scale-out 시 인스턴스마다 캐시 따로 (stale 가능)
**되돌리기 비용**: 중간

## ADR-001 — 2026-04 (초기)
**결정**: 저장소 모드 (`file` vs `jdbc`) 분리
**대안**: jdbc only
**선택 이유**:
- 로컬 개발 시 Postgres 없이 돌릴 수 있음
- AuthController 등은 `@Conditional(JdbcStoreCondition::class)` 로 jdbc 모드만 활성
**영향**: file 모드에선 `/auth/*` 404 — 의도적
**되돌리기 비용**: 중간 (jdbc only 로 단순화 가능)
