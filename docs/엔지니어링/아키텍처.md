## 패키지 구조

```
com.giwon.signaldesk/
├─ bootstrap/                       # 셸 (Health / Cors / RateLimit / Security / App entry)
├─ features/
│  ├─ market/                       # 시장 데이터
│  │  ├─ application/               # service + 외부 client
│  │  ├─ presentation/              # controller + ApiResponse wrapper
│  │  └─ ...
│  ├─ workspace/                    # 사용자 데이터 (CRUD)
│  ├─ auth/                         # 인증 (Email / Google / Kakao)
│  ├─ realtime/                     # WS price ticker
│  ├─ push/                         # Expo push
│  ├─ fortune/                      # 오늘의 운
│  └─ kakao/                        # KakaoTalk 챗봇 (별도 webhook)
└─ ...
```

## 캐싱 전략

(stub)
- `MarketOverviewService` 안에서 `cachedCore` (60s OPEN / 3m 휴장) + `cachedNews` (5m / 15m)
- `NaverStockSearchClient` 자체 ConcurrentHashMap (10분)
- 모두 in-memory. 단일 인스턴스 전제.

## 동시성

(stub)
- core cache 빌드 시 `synchronized` 단일 진입
- 외부 client 호출은 `CompletableFuture.supplyAsync` 로 병렬
- WS sessions / subscriptions: ConcurrentHashMap

## 저장소 (persistence)

(stub)
- Mode: `signal-desk.store.mode` (file / jdbc)
- file: `./data/signal-desk-store.json` (개발용)
- jdbc: PostgreSQL (Spring JDBC + Flyway like 마이그레이션 부팅 시 실행)
- repository 인터페이스: `SignalDeskWorkspaceRepository`

## 핵심 결정

(stub)
- 왜 캐시 60s: KRX/Naver 보호 + 5s WS 가 별도 채워줌
- 왜 휴장 모드 cache 늘림: 외부 호출 불필요 시간대
- 왜 file mode 도 지원: 로컬 개발 / DB 없는 환경
- 왜 OAuth audience 화이트리스트: 다른 클라이언트 ID 토큰 차단
- 왜 single instance 가정: 사용자 작은 단계 — scale-out 시 외부 cache (Redis) 필요
