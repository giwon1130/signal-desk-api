# 장애 대응 (백엔드 SoT)

> **누가 봄**: 운영자
> **언제 봄**: 장애 발생 시
> **공유**: app 저장소 [장애대응](https://github.com/giwon1130/signal-desk-app/tree/main/docs/운영/장애대응.md) 와 동일한 내용 + 백엔드 보충.

## 빠른 진단 명령

```bash
# 헬스
curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" \
  https://signal-desk-api-production.up.railway.app/health

# 로그 (최근)
railway logs --service signal-desk-api | tail -50

# 빌드 로그
railway logs --service signal-desk-api --build | tail -30

# 환경변수 확인
railway variables --service signal-desk-api --json | jq '.SIGNAL_DESK_CORS_ALLOWED_ORIGIN_PATTERNS, .SIGNAL_DESK_STORE_MODE'
```

## 상황별

### 🔴 `/health` 200 아님

(stub)
1. `railway logs --service signal-desk-api` 마지막 50줄
2. 흔한 원인:
   - `OutOfMemoryError` → 메모리 부족
   - `Connection refused: postgres` → DB 끊김
   - `BindException: Address already in use` → 포트 충돌
3. 빠른 조치: 대시보드 → Redeploy

### 🟡 외부 API 실패 폭증

(stub)
- 로그 키워드: `... fetch failed`
- 영향: 시장 요약 비어있음 — 사용자에겐 빈 카드 / "수집 중"
- 우리가 할 거: KRX/Naver/FRED 회복 대기. 더 깊게 — `signal-desk.integrations.*.enabled=false` 로 임시 차단 가능.

### 🟡 Postgres 끊김

(stub)
- 영향: `/api/v1/workspace/*` 만 500
- 시장 요약은 메모리 캐시 + 외부 데이터로 살아있음
- 조치: Railway Postgres 서비스 restart

### 🟡 CORS 403

(stub)
- 원인: `SIGNAL_DESK_CORS_ALLOWED_ORIGIN_PATTERNS` 에 도메인 누락
- 조치: env 추가 → 자동 재시작

### 🔴 인증 모두 실패

(stub)
- JWT secret 변경 직후라면 — 모든 사용자 재로그인 (예상된 동작)
- audience 화이트리스트 변경 직후라면 — Google 클라이언트 ID 누락 확인
- 그 외: DB 인증 테이블 손상 가능성 (백업 복구)

### 🟡 Rate limit 오작동

(stub)
- 정상 사용자도 429 → corporate NAT 또는 한 IP 다수 사용자
- 조치: `RateLimitFilter.kt` 의 `MARKET_RULE.capacity` 상향
- 또는: token bucket key 를 IP+JWT 조합으로 (인증된 사용자는 별도 카운트)

### 🟡 빌드 실패 (CI)

(stub)
- 보통 dependency 또는 Kotlin 컴파일 에러
- 로그: `railway logs --service signal-desk-api --build`
- 로컬 재현: `./gradlew compileKotlin test`
